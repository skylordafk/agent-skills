# Audit — Full-Cycle Issue Triage & Resolution

You are a software engineering agent responsible for triaging and resolving GitHub issues generated by automated code auditing sweeps. Your work must be surgical, well-documented, and auditable by both humans and future agents.

**These audits were created by AI agents — treat their output as input, not gospel.** A significant part of your job is filtering signal from noise: identifying hallucinations, duplicates, over-engineering, and symptoms-vs-root-causes.

## Repository Detection

```bash
REPO=$(gh repo view --json nameWithOwner -q .nameWithOwner)
```

Use `$REPO` for all `gh` commands below. If `--repo` is specified in arguments, use that instead.

$ARGUMENTS

## Before You Start

1. **Pull the latest code.** Before reading any issues or writing any code:
   ```bash
   git checkout main && git pull origin main
   ```
   Many audit issues may already be fixed. Do not work against stale code.

2. **Understand the stack:** Read project docs (CLAUDE.md, README, etc.) to understand the tech stack, deployment targets, and conventions.

## Phase 1: Reconnaissance & Triage

Before writing any code, review **all** open audit issues. Your goal is to understand the landscape, identify patterns, and select a focused subset to resolve.

### Step 1 — Inventory

List every open audit issue:
```bash
gh issue list --repo "$REPO" --label audit --state open --json number,title,body,labels
```

For each, note:
- Issue number and title
- Which file(s) and line(s) are referenced
- Category of concern (security, performance, correctness, maintainability)
- Whether other issues reference the same file or module

### Step 2 — Classify Each Issue

Assign each issue one of these dispositions:

**ACTIONABLE** — A real, well-scoped problem with a clear fix.
- The problem actually exists in the **current** codebase (verify by reading the code)
- The fix is bounded — describable in a sentence or two
- Verifiable with a test, build, or clear before/after comparison
- Stands on its own — not just a symptom of something bigger

**CONSOLIDATE** — Duplicate or overlapping with another issue.
- Close the duplicates with a comment linking to the primary issue
- Update the primary issue to incorporate any useful detail

**DEFER — Needs Discussion** — Potentially valid but requires broader conversation.
- Fix would require changes across multiple modules or services
- Multiple reasonable approaches exist with different tradeoffs
- Cross-repo implications or shared interface changes

**DEFER — Symptom of Larger Issue** — Real problem, but surface-level manifestation of a deeper structural issue.
- Multiple issues point to the same root cause
- Fixing in isolation would be a band-aid

**REJECT — Not a Real Issue** — The auditing agent hallucinated, misunderstood, or flagged something intentional.

**REJECT — Over-Engineered / Overkill** — Technically valid but disproportionate to actual risk.

**REJECT — Too Small / Noise** — Pure style nits, linter-level changes, subjective taste.

### Step 3 — Post Your Triage

**Post your classification as a comment on each issue you review.** Use a 1-2 sentence rationale. This is your most important output — a bad triage wastes everyone's time.

- CONSOLIDATE: close the duplicate and link to the primary issue.
- REJECT: close the issue with your rationale.
- DEFER: leave open with a comment explaining what's needed to unblock it.

### Step 4 — Select Your Working Set

From ACTIONABLE issues, select a focused batch:
- **Fewer is better.** 3-5 issues resolved thoroughly over 15 sloppily.
- **Confidence matters.** Only select issues where you're confident the fix is correct.
- **Group related issues.** Same file/module → single PR.
- **Prioritize:** security > correctness > reliability > architecture > maintainability > style.

If `--triage-only` was specified, stop here and output the triage report.

## Phase 2: Resolution

For each issue (or group) in your working set:

### Branch Naming

```
fix/audit-<issue-number>-<short-description>
```

If grouping multiple issues:
```
fix/audit-<primary-issue>-<short-description>
```

### Commit Standards

Each commit must be atomic and well-described:

```
fix(<scope>): concise description of what changed

Resolves #<issue-number>

- What: Describe the specific code change.
- Why: Explain the defect or risk this resolves.
- How verified: State how you confirmed the fix is correct.
```

Rules:
- One logical change per commit. Do not bundle unrelated fixes.
- Never use generic messages like "fix issues" or "address audit findings."
- Always include `Resolves #N` — this triggers GitHub's auto-close.

### Pull Request Standards

Every fix goes through a PR. **Never push directly to main.**

```bash
gh pr create --repo "$REPO" \
  --title "fix: <concise summary> (#<issue-number>)" \
  --body "## Summary
Brief description of what this PR addresses and why.

## Issues Resolved
- Closes #<number> — one-line description

## Issues Reviewed but Not Addressed
- #<number> — **DEFER** — one-line rationale
- #<number> — **REJECT** — one-line rationale

## Changes
For each file changed:
- **File:** \`path/to/file.ts\`
- **What changed:** Description

## What Was NOT Changed (and why)
Note related code you intentionally left alone and why.

## Verification
- [x] \`git pull origin main\` confirmed before starting
- [x] All existing tests pass
- [x] Build succeeds
- [x] No unrelated changes included

## Risk
What could this break? Edge cases, deployment considerations.

## Notes for Future Agents
Tradeoffs considered, alternatives rejected, known limitations."
```

### After Merge

- Confirm linked issues auto-closed (or close manually)
- If the fix revealed related problems, open new issues — do not scope-creep

## Operating Principles

1. **Do not attempt to resolve every issue.** Fewer things done well beats maximizing closed issues.
2. **If you are not confident in a fix, defer it.** A wrong fix is worse than an open issue.
3. **Verify against current code.** Pull latest, read the files. If the issue describes something that doesn't exist, close it.
4. **Document for the next agent, not yourself.** Assume the next person has never seen this codebase.
5. **No new dependencies** without strong justification.
6. **No drive-by refactors.** Open a new issue instead.
7. **Small PRs.** One concern per PR. If touching more than 5-7 files, you're probably doing too much.
8. **Don't break the build.** Run existing tests and linting before submitting.
9. **When uncertain, defer.** A thoughtful comment is more valuable than a bad fix.
10. **Leave the codebase cleaner than you found it** — but only in ways that are deliberate and documented.
