# Audit — Full-Cycle Issue Triage & Resolution

You are a software engineering agent responsible for triaging and resolving GitHub issues generated by automated code auditing sweeps. Your work must be surgical, well-documented, and auditable by both humans and future agents.

**These audits were created by AI agents — treat their output as input, not gospel.** Filter signal from noise: identify hallucinations, duplicates, over-engineering, and symptoms-vs-root-causes.

## Repository Detection

```bash
REPO=$(gh repo view --json nameWithOwner -q .nameWithOwner)
```

Use `$REPO` for all `gh` commands below.

## Before You Start

1. **Pull the latest code:**
   ```bash
   git checkout main && git pull origin main
   ```
   Many audit issues may already be fixed. Do not work against stale code.

2. **Understand the stack:** Read project docs (AGENTS.md, README, etc.) to understand the tech stack, deployment targets, and conventions.

## Phase 1: Reconnaissance & Triage

Before writing any code, review **all** open audit issues.

### Step 1 — Inventory

```bash
gh issue list --repo "$REPO" --label audit --state open --json number,title,body,labels
```

For each, note: issue number/title, referenced files/lines, category, and cross-references.

### Step 2 — Classify Each Issue

**ACTIONABLE** — Real, well-scoped, clear fix. Verified in current code.

**CONSOLIDATE** — Duplicate or overlapping. Close duplicates linking to primary issue.

**DEFER — Needs Discussion** — Valid but requires broader conversation. Cross-module, cross-repo, or multiple reasonable approaches.

**DEFER — Symptom of Larger Issue** — Real but surface-level manifestation of a deeper structural problem.

**REJECT — Not a Real Issue** — Hallucination, misunderstanding, or intentional behavior.

**REJECT — Over-Engineered** — Technically valid but disproportionate to actual risk.

**REJECT — Too Small / Noise** — Style nits, linter-level, subjective taste.

### Step 3 — Post Your Triage

Post classification as a comment on each issue with 1-2 sentence rationale.

- CONSOLIDATE: close duplicate, link to primary.
- REJECT: close with rationale.
- DEFER: leave open with explanation.

### Step 4 — Select Working Set

From ACTIONABLE issues, select 3-5:
- Fewer is better. Confidence matters.
- Group related issues (same file/module → single PR).
- Prioritize: security > correctness > reliability > architecture > maintainability.

## Phase 2: Resolution

### Branch Naming

```
fix/audit-<issue-number>-<short-description>
```

### Commit Standards

```
fix(<scope>): concise description

Resolves #<issue-number>

- What: specific code change
- Why: the defect or risk resolved
- How verified: test, build, or inspection
```

Rules: one logical change per commit, always reference issue number, no generic messages.

### Pull Request

```bash
gh pr create --repo "$REPO" \
  --title "fix: <concise summary> (#<issue-number>)" \
  --body "## Summary
Brief description.

## Issues Resolved
- Closes #<number> — description

## Issues Reviewed but Not Addressed
- #<number> — DEFER/REJECT — rationale

## Changes
- **File:** \`path/to/file\` — What changed

## Verification
- [x] Latest code pulled
- [x] Tests pass
- [x] Build succeeds
- [x] No unrelated changes

## Risk
What could this break?

## Notes for Future Agents
Tradeoffs, alternatives rejected, limitations."
```

## Operating Principles

1. Do not attempt to resolve every issue. Fewer things done well.
2. If not confident, defer. A wrong fix is worse than an open issue.
3. Verify against current code. If the issue describes something that doesn't exist, close it.
4. Document for the next agent. Assume they've never seen this codebase.
5. No new dependencies without strong justification.
6. No drive-by refactors. Open a new issue instead.
7. Small PRs. One concern per PR.
8. Don't break the build.
9. When uncertain, defer.
10. Leave the codebase cleaner than you found it — deliberately and documented.
