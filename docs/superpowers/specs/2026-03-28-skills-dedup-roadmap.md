# Agent Skills Deduplication & Robustness Roadmap

**Date:** 2026-03-28
**Scope:** github-ops skills (audit, triage, fix-issues, review-pr) across Claude Code and Codex

## Problem

The 8 SKILL.md files (4 per platform) repeat the same frameworks, templates, and conventions inline. This causes:
- Drift between copies (e.g., Claude fix-issues has emoji in Four Dimensions, Codex doesn't)
- Maintenance burden when updating shared content
- Hardcoded `npm run build` breaks cross-project portability
- No label verification causes silent failures in new repos

## Design

### 1. Shared References Directory

Create `github-ops/references/` with canonical versions of duplicated content:

```
github-ops/
├── references/
│   ├── severity-definitions.md
│   ├── four-dimensions.md
│   ├── label-taxonomy.md
│   ├── commit-and-pr-format.md
│   └── operating-principles.md
├── claude/
└── codex/
```

**How referencing works:** Each SKILL.md replaces its inline copy with a line like:

> For severity criteria, read `../../references/severity-definitions.md`

The path is relative to the SKILL.md's location in the `github-ops/` directory tree (e.g., from `github-ops/claude/triage/SKILL.md`, `../../references/` resolves to `github-ops/references/`). The agent uses its Read tool to load the file on demand. This works regardless of how skills are installed — the `github-ops/` directory structure just needs to be preserved.

### Bootstrap

Verify labels exist before starting triage or audit work:

```bash
REPO=$(gh repo view --json nameWithOwner -q .nameWithOwner)
for label in "needs-triage:FBCA04" "ready-to-fix:0E8A16" "in-progress:1D76DB" \
             "deferred:C2E0C6" "audit-resolved:006B75" "audit-rejected:D93F0B" \
             "audit:EDEDED"; do
  name="${label%%:*}" color="${label#*:}"
  gh label create "$name" --repo "$REPO" --color "$color" 2>/dev/null || true
done
```
```

#### `commit-and-pr-format.md`

Canonical source. Currently duplicated in:
- Claude: fix-issues (lines 51-57, 73-93), audit (lines 119-135, 141-173)
- Codex: fix-issues (lines 44-50, 66-86), audit (lines 85-97, 99-127)

Note: audit has a more detailed PR template than fix-issues. The canonical version uses the audit (fuller) variant. fix-issues can reference it and note which sections are optional for smaller PRs.

Content:
```markdown
## Commit Format

Each commit must be atomic — one logical change:

```
fix(<scope>): concise description

Resolves #<issue-number>

- What: specific code change
- Why: the defect or risk resolved
- How verified: test, build, or inspection
```

Rules:
- One logical change per commit
- Always reference the issue number
- Never use generic messages ("fix issues", "cleanup")
- Include `Resolves #N` to trigger GitHub's auto-close

## PR Template

```
## Summary
Brief description of what this PR addresses and why.

## Issues Resolved
- Closes #<number> — one-line description

## Issues Reviewed but Not Addressed
- #<number> — DEFER/REJECT — rationale

## Changes
For each file changed:
- **File:** `path/to/file` — What changed

## Verification
- [x] Latest code pulled
- [x] Tests pass
- [x] Build succeeds
- [x] No unrelated changes

## Risk
What could this break? Edge cases, deployment considerations.

## Notes for Reviewer / Future Agents
Tradeoffs considered, alternatives rejected, limitations.
```

For smaller fix PRs, the "Issues Reviewed but Not Addressed", "Risk", and "Notes" sections are optional.
```

#### `operating-principles.md`

Canonical source. Currently duplicated in:
- Claude: audit (lines 219-230, 10 items), fix-issues (lines 119-123, 4-item subset)
- Codex: audit (lines 162-173, 10 items), fix-issues (lines 108-113, 4-item subset)

Content:
```markdown
## Operating Principles

1. **Do not attempt to resolve every issue.** Fewer things done well beats maximizing closed issues.
2. **If you are not confident in a fix, defer it.** A wrong fix is worse than an open issue.
3. **Verify against current code.** Pull latest, read the files. If the issue describes something that doesn't exist, close it.
4. **Document for the next agent, not yourself.** Assume the next person has never seen this codebase.
5. **Every fix moves toward ideal state.** Say which dimension.
6. **No new dependencies** without strong justification.
7. **No drive-by refactors.** Open a new issue instead.
8. **Small PRs.** One concern per PR. If touching more than 5-7 files, you're probably doing too much.
9. **Don't break the build.** Run existing tests and linting before submitting.
10. **When uncertain, defer.** A thoughtful comment is more valuable than a bad fix.
```

### 3. Build System Detection

Replace all instances of hardcoded `npm run build` with:

```bash
if [ -f "package.json" ]; then
  npm run build
elif [ -f "Cargo.toml" ]; then
  cargo build
elif [ -f "Makefile" ]; then
  make
elif [ -f "go.mod" ]; then
  go build ./...
elif [ -f "pyproject.toml" ]; then
  pip install -e . 2>/dev/null || python -m build
else
  echo "No recognized build system. Skipping build verification."
fi
```

Affected files with inline `npm run build` command (4 total):
- Claude: fix-issues (line 66), review-pr (line 57)
- Codex: fix-issues (line 59), review-pr (line 54)

The audit skills don't have a direct build command — their PR template already says "Build succeeds" generically. The shared `commit-and-pr-format.md` reference also uses the generic "Build succeeds" wording.

The fix-issues skills also reference `npm run build` in the PR verification checklist (`` `npm run build` succeeds ``). Update this to `` Build succeeds `` when replacing with the shared PR template reference.
